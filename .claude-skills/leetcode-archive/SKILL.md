# LeetCode 笔记归档（/lcmd）

把当前对话里刚讨论完的那道力扣题整理成一个 md 文件，写进本地仓库根目录，然后 commit & push。

## 触发

用户输入 `/lcmd`（可能带参数，如 `/lcmd Two Sum` 指定题目；不带参数则归档**本次对话中最近讨论的那道题**）。

## 固定配置

- **仓库路径**：`~/xmn/LeetcodePractice`（文件放在仓库根目录）
- **代码语言**：Python
- **文件名**：`题号.标题.md`，例如 `1.Two Sum.md`（题号**不补零**，标题用题目的英文官方名，保留空格）
- **commit message**：`Add notes: 题号. 标题`，例如 `Add notes: 1. Two Sum`

## 工作流程

### 第 1 步：确定是哪道题

从对话历史里找出最近讨论的力扣题目，提取：题号、英文标题、难度、链接（`https://leetcode.com/problems/<slug>/`）。如果对话里信息不全（比如没提题号），先简短问用户确认，不要瞎编题号。

### 第 2 步：判断用户的掌握情况，决定 md 内容

这是本技能的核心判断。回看整段讨论，判断用户属于哪一类：

**情况 A —— 用户本来能独立完成部分思路**（中途卡住、或思路方向对但有偏差、或需要点拨才走通）：
归纳用户讨论过程中**踩过的所有主要思维误区**（不只是最后一个，按踩坑的时间顺序列出每一个），不要只挑一个"最核心"的就草草收尾，要覆盖所有曾经导致代码出错的关键卡点（命名重复/越界/死循环/逻辑顺序错误等都算，纯粹的代码风格小建议可以不列）。每一个误区都用**两个 Python 代码块对照**呈现：一个标 `❌ 错误写法`、一个标 `✅ 正确写法`，必要时在代码里用简短注释点明问题所在。主体是代码而非文字描述。

**情况 B —— 用户几乎无法独立完成**（基本是 Claude 从头讲解的）：
把整段讨论**归纳整理**成一份清晰的题解笔记，覆盖：思路推导、关键观察、最终解法。不要逐字复制对话，要重新组织成连贯、好复习的笔记。

如果不确定属于 A 还是 B，简短问一下用户，或按讨论中用户的参与程度判断。

### 第 3 步：提取知识点 Q&A

回看整段讨论，找出用户提出的**知识点/语法类提问**（即与当前题目算法思路无关、而是问语言特性或基础概念的问题，比如"为什么不用写 self"、"Optional[TreeNode] 是什么意思"、某个语法/内置函数的用法等）。把这些问题和对应解答整理成简洁的 Q&A 列表，放在 md 文件最前面（题目元信息之后、正文之前）。如果整段讨论没有出现这类提问，跳过这部分，不要硬凑。

### 第 4 步：生成 md 文件

按下面模板生成。无论 A/B，文件头部都包含题目元信息和知识点 Q&A（如有）；正文部分根据情况 A/B 填入对应内容。代码块统一用 Python。

参见 `references/template.md` 了解两种情况的完整 md 模板。

### 第 5 步：写入仓库并推送

```bash
cd ~/xmn/LeetcodePractice
# 把 md 写到根目录，文件名形如 "1.Two Sum.md"
git add "1.Two Sum.md"
git commit -m "Add notes: 1. Two Sum"
git push
```

注意：
- 文件名和 commit message 里都带空格，命令里务必加引号。
- 如果 `git push` 因为凭证 / 网络失败，把报错原样告诉用户，让用户自己处理凭证，**不要**尝试输入密码或 token。
- push 成功后，简短告诉用户：归档了哪道题、文件名、判定为情况 A 还是 B。

## 完成后的回报

一句话即可，例如："已归档 `1.Two Sum.md`（情况 A：思维误区表格），已 push 到 LeetcodePractice。"
